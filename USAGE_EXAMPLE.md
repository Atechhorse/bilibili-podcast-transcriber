## 场景 2: 中断后继续

假设处理到第 20 个链接时，你按了 Ctrl+C：

```bash
# 再次运行，会自动从第 21 个继续
python main.py

# 或明确指定 --resume
python main.py --resume
```

输出：
```
续传模式：跳过已完成的链接
[1/53] 跳过已处理: https://www.bilibili.com/video/BV14RrKBNEdH/
[2/53] 跳过已处理: https://www.bilibili.com/video/BV1H5qrByEAE/
...
[20/53] 跳过已处理: ...
[21/53] 正在下载音频: ...  ← 从这里继续
```

## 场景 3: 只重试失败的链接

如果有几个链接因为网络问题失败了：

```bash
python main.py --retry-failed
```

输出：
```
重试模式：仅处理失败的链接
找到 3 个失败的链接待重试
[1/3] 正在下载音频: ...
```

## 场景 4: 强制重新处理所有

如果你想从头开始，忽略之前的进度：

```bash
python main.py --force
```

## 场景 5: 处理其他链接文件

```bash
python main.py --input ../new_links.txt
```

## 性能参考

基于 M1 MacBook Pro (16GB RAM):

| 项目 | 单个视频耗时 | 53个视频总耗时 |
|------|-------------|---------------|
| 元数据提取 | ~2秒 | ~2分钟 |
| 音频下载 | 30秒-2分钟 | ~1-2小时 |
| 语音转写 (30分钟音频) | ~5分钟 | ~4-5小时 |

**优化建议**:
- 可以分批处理（每次处理 10 个）
- 使用 GPU 加速会快 3-5 倍
- 可以先只做元数据提取和下载，转写可以后台慢慢跑

## 故障处理示例

### 问题1: 某个视频下载失败
```
[15/53] ✗ 下载失败: HTTP Error 403: Forbidden
```

**解决**: 
- 该视频可能有地区限制或需要登录
- 系统会继续处理其他视频
- 后续可以用 `--retry-failed` 单独重试

### 问题2: 转写时内存不足
```
[20/53] ✗ 转写失败: CUDA out of memory
```

**解决**:
```yaml
# 修改 config.yaml
whisper:
  model_size: "small"  # 从 medium 改为 small
  device: "cpu"        # 或使用 CPU
```

### 问题3: LLM API 调用失败
```
✗ 说话人身份推理失败: API rate limit exceeded
```

**影响**: 说话人标签会保留为 SPEAKER_00, SPEAKER_01 等
**解决**: 稍后重试，或暂时接受原始标签

## 高级用法

### 只提取元数据（不下载音频）

临时注释掉 `main.py` 中的步骤 2-4，只运行步骤 1。

### 批量处理策略

```bash
# 第一批：前 10 个
head -10 视频链接 > batch1.txt
python main.py --input batch1.txt

# 第二批：接下来 10 个
sed -n '11,20p' 视频链接 > batch2.txt
python main.py --input batch2.txt
```

### 只处理 B站视频

```bash
grep "bilibili" 视频链接 > bilibili_only.txt
python main.py --input bilibili_only.txt
```

**提示**: 第一次运行建议先测试 1-2 个链接，确认流程正常后再处理全部。
