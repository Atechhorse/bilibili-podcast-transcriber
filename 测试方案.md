# 视频/播客音频提取与逐字稿生成项目测试方案

## 1. 测试目标
确保系统能够稳定地从 Bilibili 和 小宇宙 平台下载音频，准确转换文字并区分说话人，最后输出符合要求的文档和表格。

## 2. 测试环境准备
-   **操作系统**: macOS (用户环境)
-   **依赖工具**: `ffmpeg` (必须安装), `Python 3.9+`
-   **Python 库**: 见 `requirements.txt`
-   **硬件**: 建议使用 M1/M2/M3 芯片 Mac 以加速推理，或具备 NVIDIA GPU 的机器。

## 3. 单元测试计划

### 3.1 链接解析测试 (`test_parser.py`)
-   **Case 1: 混合输入**
    -   输入：包含 Bilibili（带/不带 https）、小宇宙、空行、乱码的文本。
    -   预期：成功提取所有有效链接，自动补全协议头，去除重复项。
-   **Case 2: 边界条件**
    -   输入：空文件。
    -   预期：返回空列表，不报错。

### 3.2 链接元数据提取测试 (`test_metadata_extractor.py`) ← 新增
> **对应需求4**: 把视频链接中的所有链接反向提取出来，整理成表格。

-   **Case 1: Bilibili 元数据提取**
    -   输入：一个 Bilibili 视频 URL。
    -   预期：返回包含 BV号、标题、UP主、发布时间、时长 的字典。
-   **Case 2: 小宇宙元数据提取**
    -   输入：一个小宇宙 URL。
    -   预期：返回包含 EpisodeID、播客名、单集标题、时长 的字典。
-   **Case 3: 批量提取生成表格**
    -   输入：包含 3 个混合链接的列表。
    -   预期：生成 `links_summary.xlsx`，包含正确的列和数据。

### 3.3 下载模块测试 (`test_downloader.py`)
-   **Case 1: Bilibili 下载**
    -   输入：一个短视频的 URL（测试用）。
    -   预期：`downloads/` 目录下生成对应的音频文件（.mp3/.m4a），文件大小 > 0。
-   **Case 2: 小宇宙下载**
    -   输入：一个小宇宙单集 URL。
    -   预期：成功解析音频流地址并下载文件。
-   **Case 3: 无效链接**
    -   输入：不存在的视频 URL。
    -   预期：捕获异常，记录错误日志，程序不崩溃。

### 3.4 转写与分离测试 (`test_transcriber.py`)
*注意：由于模型运行较慢，此部分建议使用裁剪后的 30秒 短音频进行测试。*
-   **Case 1: 说话人区分**
    -   输入：一段双人对话的测试音频。
    -   预期：输出结果包含至少两个不同的 speaker_id。
-   **Case 2: 身份识别逻辑**
    -   **单人测试**: 输入单人独白音频，预期输出 Speaker 为 “付鹏”。
    -   **多人测试**: 输入一段包含自我介绍（“我是付鹏”）的对话音频，预期程序能通过前几分钟的文本分析，将对应的 Speaker ID 自动重命名为 “付鹏”。
-   **Case 3: 转写准确性**
    -   输入：标准普通话测试音频。
    -   预期：生成的文本包含预期的关键词。
-   **Case 4: LLM 身份推理 Mock 测试** ← 新增
    -   使用 Mock 替换真实 LLM 调用，返回固定的身份推理结果。
    -   验证程序能正确解析 LLM 返回结果并替换 Speaker ID。

### 3.5 报告生成模块测试 (`test_reporter.py`) ← 新增
-   **Case 1: 逐字稿 Markdown 生成**
    -   输入：模拟的转写结果数据（包含时间戳、说话人、文本）。
    -   预期：生成格式正确的 Markdown 文件，包含标题、元信息、逐字稿。
-   **Case 2: 汇总表格更新**
    -   输入：多个处理结果。
    -   预期：正确更新 `links_summary.xlsx`，新增状态列。

### 3.6 进度管理模块测试 (`test_progress_manager.py`) ← 新增
-   **Case 1: 进度保存与恢复**
    -   模拟处理 3 个链接，中途中断。
    -   重启后读取进度文件，验证已完成的任务被跳过。
-   **Case 2: 状态更新**
    -   验证每个阶段完成后，`progress.json` 中对应链接的状态正确更新。
-   **Case 3: --retry-failed 参数**
    -   模拟 1 个失败任务，使用 `--retry-failed` 运行，验证只重试失败项。

## 4. 集成测试 (End-to-End)

### 4.1 流程测试
-   **步骤**:
    1.  创建一个包含 1 个 B站链接和 1 个小宇宙链接的 `test_links.txt`。
    2.  运行主程序 `main.py`。
-   **验收标准**:
    -   `downloads/` 文件夹中有 2 个音频文件。
    -   `transcripts/` 文件夹中有 2 个对应的 Markdown 文件。
    -   Markdown 内容包含时间戳和 Speaker 标识。
    -   根目录下生成 `summary.csv`，包含正确的反向提取链接。

### 4.2 断点续传测试 ← 新增
-   **步骤**:
    1.  准备 5 个链接的测试文件。
    2.  运行程序，在处理到第 3 个时手动中断（Ctrl+C）。
    3.  再次运行程序（默认 `--resume` 模式）。
-   **验收标准**:
    -   程序从第 3 个链接继续处理，不重复处理前 2 个。
    -   最终所有 5 个链接都处理完成。

### 4.3 长音频分段测试 ← 新增
-   **步骤**:
    1.  准备一个超过 30 分钟的测试音频（可使用小宇宙长播客）。
    2.  运行转写流程。
-   **验收标准**:
    -   `data/temp/` 中生成分段文件，处理完后自动清理。
    -   最终逐字稿时间戳连贯，无明显断裂或重复。

### 4.4 性能与资源监控
-   运行过程中观察内存和 CPU 占用。
-   确保在处理长音频（>1小时）时不会发生 OOM (内存溢出)。

## 5. 验收数据 (User Acceptance Data)
使用用户提供的 `视频链接` 文件的前 3 行作为验收测试集，验证最终效果。

```text
1. https://www.bilibili.com/video/BV14RrKBNEdH/...
2. bilibili.com/video/BV1H5qrByEAE/...
3. https://www.xiaoyuzhoufm.com/episode/69377c0c3fec3166cfff72fd
```

## 6. 风险与应对
-   **反爬虫风险**: B站和小宇宙可能有反爬策略。
    -   *应对*: 使用 `yt-dlp` 的 cookie 加载功能或降低请求频率。
-   **模型幻觉**: Whisper 可能在静音段输出无意义文本。
    -   *应对*: 使用 VAD (Voice Activity Detection) 预处理，过滤静音片段。
